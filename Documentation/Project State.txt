Application entry flow:
main.dart initializes Flutter bindings, attempts to load .env, initializes Firebase, optionally initializes Supabase if environment values are present, and then launches ArtflowApp.

Global app composition:
app.dart creates and provides AuthState and AppDataState, synchronizes shared app data after auth changes, applies the global theme, and attaches the router.

Authenticated admins are redirected toward /admin.
Verified artists are allowed to access create and artist dashboard routes.
Non-artist users are blocked from artwork creation routes.

Backend and state architecture summary:

Security and backend notes:
firestore.rules exists in the repository and covers users, artworks, commissions, orders, notifications, reviews, transactions, disputes, artist applications, and platform settings.
The project currently creates some cross-user notifications directly from the client, which works for the current app flow but should eventually move to trusted backend logic for stronger production security.

Overall current state:
The project is already functioning as a Firebase-backed Flutter marketplace application with buyer, artist, and admin flows. Authentication, profile persistence, artist application handling, realtime chat, artwork management, commissions, orders, notifications, mock checkout, and a broad admin module are present. The biggest next improvements would be splitting the large screen file, replacing the mock payment flow with a real provider, strengthening backend-driven security behaviors, and expanding automated test coverage.
